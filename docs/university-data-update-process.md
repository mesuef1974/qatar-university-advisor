# عملية تحديث بيانات الجامعات

**الإصدار:** 1.0

**المسؤول:** فريق تطوير Qatar University Advisor

## 1. Overview (نظرة عامة)

هذا الدليل يشرح العملية الكاملة لتحديث بيانات الجامعات في التطبيق. يجب تنفيذ هذه العملية سنوياً قبل كل موسم قبول جامعي للتأكد من دقة المعلومات المقدمة للطلاب.

### الملفات التي تحتاج إلى تحديث

| الملف | الغرض |
|-------|--------|
| `lib/data/universities-2026.js` | البيانات الأساسية للجامعات (معدلات القبول، الرسوم، المنح) |
| `lib/responses.js` | ردود الذكاء الاصطناعي الأساسية المرتبطة بالبيانات |
| `lib/responses-extended.js` | الردود الموسعة والتفصيلية للاستفسارات المعقدة |

### توقيت التحديث

- **الموعد المثالي:** مارس - أبريل من كل عام
- **السبب:** يتزامن مع فتح بوابات القبول الجامعي في قطر
- **التكرار:** مرة واحدة سنوياً كحد أدنى، مع مراجعة نصف سنوية للتغييرات الطارئة

### مصادر البيانات الرئيسية

- المواقع الرسمية لكل جامعة (قسم القبول والتسجيل)
- وزارة التعليم والتعليم العالي — MoEHE Qatar: [moehe.gov.qa](https://www.moehe.gov.qa)

## 3. Update Checklist (قائمة التحقق)

اتبع هذه الخطوات بالترتيب عند تنفيذ التحديث السنوي. ضع علامة على كل خطوة بعد إتمامها.

### مرحلة جمع البيانات

- [ ] التحقق من متطلبات المعدل التراكمي (GPA) لكل جامعة — الطلاب القطريون وغير القطريين
- [ ] تحديث مواعيد التقديم للفصل الدراسي الأول (Fall) والثاني (Spring)
- [ ] تحديث الرسوم الدراسية (Tuition Fees) لكل جامعة
- [ ] تحديث قيم المنح الدراسية وشروطها
- [ ] التحقق من البرامج الأكاديمية الجديدة أو المحذوفة
- [ ] تحديث حالة Texas A&M-Q (مغلقة للقبول — تغلق نهائياً 2028)
- [ ] التحقق من تفاصيل المنح الدراسية الرئيسية:
  - منحة أميري (Amiri Scholarship)
  - منحة تميّح (Tamuh Scholarship)
  - المنح المؤسسية (Corporate Scholarships)

### مرحلة تحديث الكود

- [ ] تحديث `lib/data/universities-2026.js` بالبيانات الجديدة
- [ ] مراجعة وتحديث الردود ذات الصلة في `lib/responses.js`
- [ ] مراجعة وتحديث الردود الموسعة في `lib/responses-extended.js`
- [ ] تشغيل الاختبارات للتأكد من عدم وجود أخطاء:

```bash
npm test
```

- [ ] اختبار 5 استفسارات شائعة يدوياً عبر واجهة الدردشة (راجع القسم 4)
- [ ] التحقق من توليد تقرير PDF بالبيانات المحدثة

### مرحلة توثيق التغييرات

- [ ] تحديث اسم ملف البيانات إن لزم (مثال: `universities-2027.js` للسنة القادمة)
- [ ] Commit التغييرات بالرسالة الموحدة:

```bash
git add lib/data/universities-2026.js lib/responses.js lib/responses-extended.js
git commit -m "data(universities): تحديث بيانات [YEAR]"
```

- [ ] تحديث هذا الملف بتاريخ آخر تحديث

## 4. Verification (التحقق)

بعد تطبيق التحديثات، يجب التحقق من صحة البيانات قبل النشر في بيئة الإنتاج.

### 4.1 تشغيل مجموعة الاختبارات

```bash
npm test
```

جميع الاختبارات يجب أن تنجح (0 failures) قبل المتابعة.

### 4.2 اختبار 5 استفسارات شائعة يدوياً

اختبر الاستفسارات التالية عبر واجهة الدردشة وتحقق من دقة الردود:

1. "ما هو المعدل المطلوب للقبول في جامعة قطر؟"
2. "ما هي المنح الدراسية المتاحة في Education City؟"
3. "ما هي الرسوم الدراسية في جامعة Carnegie Mellon Qatar؟"
4. "هل Texas A&M القطر لا تزال تقبل طلبات؟"
5. "ما هي مواعيد التقديم لجامعة UDST؟"

### 4.3 التحقق من توليد تقرير PDF

- فتح ميزة تقرير PDF في التطبيق
- التأكد من أن التقرير يعكس البيانات المحدثة
- التحقق من عدم وجود بيانات مكررة أو متناقضة

### 4.4 معايير القبول للنشر

| المعيار | الحد المطلوب |
|---------|-------------|
| نجاح الاختبارات | 100% |
| دقة الاستفسارات اليدوية | 5/5 صحيح |
| توليد PDF | يعمل بدون أخطاء |

## 5. Automation Roadmap (خارطة أتمتة التحديث)

الهدف على المدى البعيد هو أتمتة عملية التحديث لتقليل الجهد اليدوي وضمان دقة البيانات في الوقت الفعلي.

### Phase 1 — الحالي: تحديث يدوي موثق

**الوضع الراهن**

- عملية يدوية بالكامل يقوم بها فريق التطوير
- موثقة بهذا الدليل لضمان الاتساق
- تكرار سنوي مع مراجعات نصف سنوية

**المزايا:** بسيطة، موثوقة، لا تتطلب بنية تحتية إضافية
**العيوب:** تستهلك وقت الفريق، احتمال تأخر التحديثات

### Phase 2 — المستقبل القريب: نصف آلي

**الهدف:** بناء web scraper يراقب صفحات القبول الرسمية

```
المراقبة التلقائية:
├── يفحص مواقع الجامعات أسبوعياً
├── يكتشف التغييرات في المعدلات والمواعيد والرسوم
└── يرسل تنبيهات للفريق عند وجود تغييرات
```

**التقنيات المقترحة:** Python + BeautifulSoup/Playwright، GitHub Actions للجدولة، تنبيهات عبر Slack/Email

### Phase 3 — المستقبل البعيد: أتمتة كاملة

**الهدف:** تكامل مباشر مع بوابات الجامعات (إن أتاحت APIs رسمية)

```
التدفق المستهدف:
University Portal API → Data Transformer → lib/data/ → Auto PR → Review → Deploy
```

**الشروط المسبقة:** توفر APIs رسمية من الجامعات، موافقة أصحاب المصلحة، بنية تحتية للمراقبة

## ملاحظات ختامية

- **لا تعتمد على بيانات غير رسمية** — المصادر الوحيدة المقبولة هي المواقع الرسمية للجامعات و MoEHE
- **وثّق أي استثناءات** — إذا لم تتوفر بيانات جامعة معينة، سجّل ذلك في ملف `CHANGELOG.md`
- **تواصل مع المستخدمين** — عند تغيير بيانات جوهرية، أضف إشعاراً في واجهة التطبيق لفترة مؤقتة
