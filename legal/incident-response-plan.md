## 1. الغرض والنطاق — Purpose & Scope

### 1.1 الغرض
تهدف هذه الخطة إلى تحديد الإجراءات المنهجية للتعامل مع أي حادثة أمنية أو اختراق للبيانات الشخصية في تطبيق "مستشار جامعة قطر" (Qatar University Advisor). تضمن هذه الخطة الامتثال الكامل للمتطلبات القانونية والتنظيمية في دولة قطر.

### 1.2 المرجعيات القانونية
- **قانون حماية خصوصية البيانات الشخصية (PDPPL)** — القانون رقم 13 لسنة 2016 بشأن حماية خصوصية البيانات الشخصية
- **المادة 16 (Article 16)** — الإبلاغ الإلزامي عن الاختراقات خلال 72 ساعة إلى المركز الوطني للأمن السيبراني (NCSA)
- **المادة 17** — حق الأفراد في الإخطار عند تعرض بياناتهم للاختراق
- **قرارات وتوجيهات NCSA** — المعايير الفنية لحماية البيانات

### 1.3 النطاق
تنطبق هذه الخطة على:
- جميع البيانات الشخصية المعالجة في التطبيق (أسماء الطلاب، بيانات أكاديمية، معلومات التواصل)
- جميع الأنظمة والخدمات المرتبطة: Supabase، Vercel، Cloudflare، Meta APIs، Google APIs
- جميع أعضاء الفريق والمتعاقدين الذين لديهم صلاحية الوصول إلى البيانات
- البيئات: Production، Staging، Development

## 2. تصنيف الحوادث — Incident Classification

### المستوى 1 — منخفض الخطورة (Severity: Low)
| البند | التفاصيل |
|---|---|
| **التعريف** | حادثة أمنية محدودة التأثير بدون تسريب فعلي للبيانات الشخصية |
| **الأمثلة** | محاولة وصول غير مصرح بها فاشلة، فحص منافذ (Port Scanning)، رسائل Phishing لم يتم النقر عليها، خطأ في إعدادات الصلاحيات تم اكتشافه داخلياً قبل الاستغلال |
| **وقت الاستجابة** | خلال 24 ساعة عمل |
| **الإبلاغ** | داخلي فقط — Security Lead + DevOps |
| **متطلبات NCSA** | لا يتطلب إبلاغ NCSA |

### المستوى 2 — متوسط الخطورة (Severity: Medium)
| البند | التفاصيل |
|---|---|
| **التعريف** | وصول غير مصرح به مؤكد أو تسريب محدود لبيانات غير حساسة |
| **الأمثلة** | تسريب عناوين بريد إلكتروني لعدد محدود من المستخدمين (أقل من 100)، اختراق حساب مشرف واحد بدون وصول لبيانات المستخدمين، خلل في API يكشف بيانات غير حساسة، Malware على جهاز موظف بدون وصول للـ Production |
| **وقت الاستجابة** | خلال 4 ساعات |
| **الإبلاغ** | Incident Commander + Security Lead + Legal + CEO |
| **متطلبات NCSA** | تقييم الحاجة للإبلاغ — يتم الإبلاغ إذا تأثرت بيانات شخصية |

### المستوى 3 — عالي الخطورة / حرج (Severity: Critical)
| البند | التفاصيل |
|---|---|
| **التعريف** | تسريب واسع للبيانات الشخصية أو اختراق كامل للنظام |
| **الأمثلة** | تسريب قاعدة بيانات المستخدمين كاملة، هجوم Ransomware على البنية التحتية، اختراق Supabase أو Vercel يؤثر على جميع المستخدمين، تسريب بيانات أكاديمية حساسة (GPA، سجلات أكاديمية)، استغلال SQL Injection ناجح |
| **وقت الاستجابة** | فوري — خلال 30 دقيقة |
| **الإبلاغ** | جميع أعضاء فريق الاستجابة + NCSA خلال 72 ساعة + المستخدمون المتأثرون |
| **متطلبات NCSA** | إبلاغ إلزامي خلال 72 ساعة وفقاً للمادة 16 من PDPPL |

## 3. الأدوار والمسؤوليات — Roles & Responsibilities Matrix

| الدور | المسؤوليات |
|---|---|
| **قائد الحادثة (Incident Commander)** | القيادة العامة للاستجابة، اتخاذ القرارات، تنسيق الفرق، التصعيد، تحديد مستوى الخطورة، الموافقة على الاتصالات الخارجية |
| **قائد الأمن (Security Lead)** | التحقيق الفني، تحليل الأدلة الرقمية (Digital Forensics)، تحديد نطاق الاختراق، احتواء التهديد، تأمين الأنظمة |
| **القسم القانوني (Legal / DPO)** | تقييم الالتزامات القانونية، إعداد تقارير NCSA، صياغة إشعارات المستخدمين، التواصل مع الجهات التنظيمية، حفظ الأدلة القانونية |
| **المدير التنفيذي (CEO)** | الموافقة النهائية على الاتصالات العامة، القرارات الاستراتيجية، التواصل مع الشركاء والمستثمرين |
| **فريق DevOps** | عزل الأنظمة المتأثرة، استعادة النسخ الاحتياطية، تطبيق الإصلاحات الفنية، مراقبة الأنظمة بعد الاستعادة |
| **فريق الاتصالات (Communications)** | صياغة البيانات الصحفية، إدارة وسائل التواصل الاجتماعي، التواصل مع المستخدمين، إدارة السمعة |

### مصفوفة التصعيد — Escalation Matrix
```
المستوى 1: Security Lead → DevOps (خلال 24 ساعة)
المستوى 2: Incident Commander → Security Lead → Legal → CEO (خلال 4 ساعات)
المستوى 3: جميع الأدوار فوراً (خلال 30 دقيقة) + NCSA (خلال 72 ساعة)
```

## 4. إجراءات الكشف — Detection Procedures

### 4.1 المراقبة الآلية (Automated Monitoring)
- **Vercel Logs** — مراقبة أنماط الطلبات غير الطبيعية (Anomalous Request Patterns)
- **Supabase Alerts** — تنبيهات الوصول غير المصرح به، محاولات تسجيل الدخول الفاشلة المتكررة
- **Cloudflare WAF** — تنبيهات هجمات DDoS وSQL Injection وXSS
- **Sentry** — مراقبة الأخطاء البرمجية غير المتوقعة التي قد تشير لاختراق
- **GitHub Security Alerts** — تنبيهات الثغرات في المكتبات المستخدمة (Dependency Vulnerabilities)

### 4.2 بلاغات المستخدمين (User Reports)
- نموذج الإبلاغ عن مشكلة أمنية في التطبيق
- البريد الإلكتروني الأمني: security@[domain].qa
- تقارير عبر وسائل التواصل الاجتماعي

### 4.3 الكشف الداخلي (Internal Discovery)
- مراجعات الكود الأمنية (Security Code Reviews)
- اختبارات الاختراق الدورية (Penetration Testing)
- تدقيق صلاحيات الوصول الشهري (Access Audit)
- فحص السجلات اليدوي (Manual Log Review)

### 4.4 مصادر خارجية (External Sources)
- تنبيهات NCSA وQ-CERT
- تقارير الباحثين الأمنيين (Bug Bounty / Responsible Disclosure)
- تنبيهات مزودي الخدمة (Supabase، Vercel، Cloudflare)

## 6. استراتيجيات الاحتواء — Containment Strategies

### 6.1 الاحتواء الفوري (Short-term Containment)
- **عزل الشبكة**: فصل الأنظمة المتأثرة عن الشبكة فوراً
- **تعليق الحسابات**: تعطيل جميع الحسابات المشبوهة أو المخترقة فوراً
- **تدوير المفاتيح**: إعادة توليد جميع API Keys وSecrets المتأثرة (Supabase، Vercel، إلخ)
- **تفعيل وضع الصيانة**: وضع التطبيق في Maintenance Mode إذا لزم الأمر
- **حظر عناوين IP**: حظر عناوين IP المشبوهة عبر Cloudflare WAF
- **تعطيل OAuth**: تعطيل تسجيل الدخول عبر Meta/Google مؤقتاً إذا كان مصدر الاختراق

### 6.2 الاحتواء طويل المدى (Long-term Containment)
- تطبيق التصحيحات الأمنية (Security Patches)
- تحديث قواعد WAF
- مراجعة وتقوية صلاحيات الوصول (Row Level Security في Supabase)
- تفعيل المصادقة الثنائية (2FA) على جميع حسابات الإدارة
- مراجعة شاملة للكود المتأثر

## 7. متطلبات حفظ الأدلة — Evidence Preservation

### 7.1 الأدلة الواجب حفظها
- سجلات الخادم (Server Logs) من Vercel وSupabase
- سجلات الوصول (Access Logs) مع الطوابع الزمنية (Timestamps)
- سجلات Cloudflare WAF
- لقطات من قاعدة البيانات (Database Snapshots)
- رسائل البريد الإلكتروني المرتبطة بالحادثة
- لقطات الشاشة والتسجيلات (Screenshots)
- نسخ من الملفات المتأثرة أو المعدلة
- سجلات Git — أي Commits مشبوهة

### 7.2 قواعد حفظ الأدلة
- **عدم التعديل**: لا تعدل أي دليل — احفظ نسخة أصلية (Read-only)
- **التوثيق الزمني**: سجل تاريخ ووقت كل إجراء بالدقيقة
- **سلسلة الحفظ (Chain of Custody)**: وثق من تعامل مع كل دليل ومتى
- **التخزين الآمن**: احفظ الأدلة في مكان منفصل ومشفر (Encrypted Storage)
- **مدة الاحتفاظ**: 3 سنوات كحد أدنى وفقاً لمتطلبات PDPPL

## 8. إجراءات الإبلاغ — Notification Procedures

### 8.1 إبلاغ المركز الوطني للأمن السيبراني (NCSA)

**الالتزام القانوني**: وفقاً للمادة 16 من قانون PDPPL (القانون 13/2016)، يجب إبلاغ NCSA خلال **72 ساعة** من اكتشاف أي اختراق يؤثر على البيانات الشخصية.

**طريقة الإبلاغ**:
1. إرسال التقرير الأولي عبر البوابة الإلكترونية لـ NCSA
2. المتابعة بتقرير تفصيلي خلال 10 أيام عمل
3. تقديم التقرير النهائي بعد إغلاق الحادثة

### 8.2 إشعار المستخدمين المتأثرين
- يتم الإشعار خلال 72 ساعة من تأكيد الاختراق
- يتم الإشعار باللغتين العربية والإنجليزية
- يشمل: طبيعة الاختراق، البيانات المتأثرة، الإجراءات المتخذة، التوصيات للمستخدم
- القنوات: بريد إلكتروني + إشعار داخل التطبيق

### 8.3 التصعيد الداخلي
```
الاكتشاف → Security Lead (فوراً)
             → Incident Commander (خلال 15 دقيقة)
             → CEO + Legal (خلال 30 دقيقة للمستوى 2+)
             → فريق الاتصالات (خلال ساعة للمستوى 3)
```

## 9. إجراءات الاستعادة — Recovery Procedures

### 9.1 استعادة الخدمة
1. التأكد من إزالة التهديد بالكامل من جميع الأنظمة
2. استعادة البيانات من آخر نسخة احتياطية سليمة (Clean Backup)
3. تطبيق جميع التصحيحات الأمنية المطلوبة
4. إعادة توليد جميع المفاتيح والأسرار (Keys Rotation)
5. اختبار شامل للنظام قبل إعادة التشغيل
6. إعادة التشغيل التدريجي (Gradual Rollout) مع مراقبة مكثفة

### 9.2 التحقق بعد الاستعادة
- فحص اختراق شامل (Penetration Test) بعد الإصلاح
- مراقبة مكثفة لمدة 72 ساعة على الأقل
- التحقق من سلامة البيانات (Data Integrity Check)
- مراجعة سجلات الوصول للتأكد من عدم وجود وصول غير مصرح به
- اختبار جميع الوظائف الحرجة في التطبيق

## 10. مراجعة ما بعد الحادثة — Post-Incident Review / Lessons Learned

### 10.1 الجدول الزمني
- **المستوى 1**: مراجعة مكتوبة خلال 5 أيام عمل
- **المستوى 2**: اجتماع مراجعة خلال 5 أيام عمل + تقرير مكتوب خلال 10 أيام
- **المستوى 3**: اجتماع مراجعة شامل خلال 7 أيام + تقرير تفصيلي خلال 14 يوم

### 10.2 محتويات تقرير المراجعة (Blameless Post-Mortem)
1. **ملخص الحادثة**: ماذا حدث؟ متى؟ ما النطاق؟
2. **الجدول الزمني التفصيلي (Timeline)**: كل إجراء بالدقيقة من الاكتشاف إلى الإغلاق
3. **تحليل السبب الجذري (Root Cause Analysis)**: لماذا حدث الاختراق؟
4. **تقييم الاستجابة**: ما الذي نجح؟ ما الذي يمكن تحسينه؟
5. **الإجراءات التصحيحية (Action Items)**: قائمة بالتحسينات المطلوبة مع مواعيد ومسؤولين
6. **تحديث الخطة**: تحديث هذه الخطة بناءً على الدروس المستفادة

### 10.3 قواعد المراجعة
- المراجعة **بدون لوم** (Blameless) — التركيز على الأنظمة لا الأشخاص
- مشاركة جميع أعضاء فريق الاستجابة
- توثيق جميع الدروس المستفادة في قاعدة المعرفة الداخلية
- متابعة تنفيذ الإجراءات التصحيحية خلال 30 يوماً

## 13. جدول الاختبارات والتدريب — Testing & Training Schedule

### 13.1 تمارين المحاكاة الربعية (Quarterly Tabletop Exercises)

| الربع | النوع | السيناريو المقترح |
|---|---|---|
| **Q1** (يناير-مارس) | Tabletop Exercise | سيناريو تسريب قاعدة بيانات Supabase كاملة |
| **Q2** (أبريل-يونيو) | Tabletop Exercise | سيناريو هجوم Ransomware على البنية التحتية |
| **Q3** (يوليو-سبتمبر) | Tabletop Exercise | سيناريو اختراق OAuth عبر Meta/Google |
| **Q4** (أكتوبر-ديسمبر) | Full Simulation | محاكاة كاملة لحادثة مستوى 3 مع إبلاغ NCSA |

### 13.2 التدريب المطلوب
- تدريب سنوي لجميع أعضاء فريق الاستجابة على هذه الخطة
- تدريب ربع سنوي على أدوات الكشف والمراقبة
- تدريب على حفظ الأدلة الرقمية (Digital Forensics Basics)
- تحديث التدريب عند أي تغيير جوهري في الخطة

### 13.3 مقاييس الأداء (KPIs)
- **MTTD** (Mean Time To Detect): هدف أقل من ساعة واحدة
- **MTTR** (Mean Time To Respond): هدف أقل من 30 دقيقة للمستوى 3
- **نسبة الامتثال لـ 72 ساعة**: 100% لجميع حوادث المستوى 3
- **نسبة إكمال التدريب**: 100% لجميع أعضاء فريق الاستجابة
