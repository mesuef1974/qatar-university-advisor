## ملاحظة قانونية هامة

> الإصدار السابق (v1.0) احتوى على ادعاء بأن "الموافقة الضمنية بالاستخدام" (implicit consent) تستوفي متطلبات المادة 7.
> **هذا الادعاء محذوف وملغى تماماً.** المادة 7 من PDPPL تتطلب موافقة صريحة ومستنيرة (explicit informed consent).
> الاعتماد على الموافقة الضمنية يشكل مخاطرة قانونية عالية وقد يعرض الشركة لعقوبات بموجب الباب السابع.

## الباب الثالث: الموافقة (Chapter 3: Consent)

### المادة 7 — الموافقة الصريحة والمستنيرة (Explicit Informed Consent)

| البند | التفاصيل |
|---|---|
| **الحالة** | ❌ غير مكتمل — خطورة عالية |
| **المتطلب** | الحصول على موافقة صريحة ومستنيرة وحرة وقابلة للسحب قبل جمع أو معالجة البيانات الشخصية |
| **التطبيق الحالي** | **لا توجد آلية موافقة صريحة.** الإصدار السابق ادعى "موافقة ضمنية بالاستخدام" وهذا غير كافٍ قانونياً |
| **الدليل/الإثبات** | لا يوجد — مطلوب بناء Consent Form |
| **الفريق المسؤول** | القسم القانوني (صياغة النص) + فريق الهندسة (التنفيذ التقني) + فريق UX (التصميم) |
| **الموعد النهائي** | **2026-05-01 — أولوية قصوى** |
| **درجة الخطورة** | **عالية (High)** |
| **الإجراء المطلوب** | 1) تصميم Consent Banner/Modal يظهر عند أول استخدام 2) النص يوضح: ما البيانات المجمعة، لماذا، مع من تُشارك، حقوق المستخدم 3) زر "أوافق" صريح + خيار "أرفض" 4) حفظ سجل الموافقة (consent log) في Supabase مع timestamp 5) آلية سحب الموافقة في أي وقت |

### المادة 8 — شروط الموافقة الخاصة (Special Consent Conditions)

| البند | التفاصيل |
|---|---|
| **الحالة** | ❌ غير مكتمل |
| **المتطلب** | موافقة إضافية مشددة لمعالجة البيانات الحساسة (صحية، دينية، عرقية) أو بيانات القاصرين |
| **التطبيق الحالي** | التطبيق لا يجمع بيانات حساسة حالياً، لكن لا يوجد فلتر تقني يمنع المستخدم من إدخالها في المحادثة |
| **الدليل/الإثبات** | لا يوجد — مطلوب تقييم |
| **الفريق المسؤول** | فريق الهندسة + قسم البيانات والذكاء الاصطناعي |
| **الموعد النهائي** | 2026-05-15 |
| **درجة الخطورة** | متوسطة (Medium) |
| **الإجراء المطلوب** | 1) إضافة PII detection filter في `ai-handler.js` لتحذير المستخدم عند إدخال بيانات حساسة 2) توثيق أن التطبيق لا يطلب بيانات حساسة بالتصميم (Privacy by Design) |
