## 1. تعريف "اختراق البيانات" في سياق هذه الخطة

وفق المادة 22 من PDPPL، يُعدّ "اختراق البيانات" أي حادثة تؤدي إلى:
- **الإفصاح غير المصرح به** عن بيانات شخصية لطرف غير مخوّل.
- **الوصول غير المصرح به** إلى بيانات شخصية، حتى دون نسخها.
- **فقدان أو تدمير** بيانات شخصية (عمداً أو بالخطأ).
- **تغيير غير مصرح به** يمس دقة البيانات الشخصية.

## 2. تصنيف الحوادث (Incident Classification)

### المستوى 1: منخفض (Low)

| المعيار | الوصف |
|---|---|
| **النطاق** | أقل من 10 مستخدمين متضررين |
| **نوع البيانات** | بيانات تقنية أو إحصائية، لا تعريفية مباشرة |
| **احتمال الضرر** | ضعيف جداً — لا تأثير يُذكر على المستخدمين |
| **المثال** | خطأ في سجلات تقنية كشف عن معرّفات UUID داخلية |
| **الإخطار الرقابي** | غير مطلوب وفق PDPPL |
| **وقت الإغلاق المستهدف** | 72 ساعة |

### المستوى 2: متوسط (Medium)

| المعيار | الوصف |
|---|---|
| **النطاق** | 10-100 مستخدم متضرر |
| **نوع البيانات** | بيانات شبه حساسة (بيانات أكاديمية دون أرقام هواتف) |
| **احتمال الضرر** | محدود — قد يسبب إحراجاً أو ضرراً طفيفاً |
| **المثال** | تسرب بيانات أكاديمية لعدد محدود من الطلاب |
| **الإخطار الرقابي** | مطلوب إذا وُجد خطر واضح على الحقوق — PDPPL م.22 |
| **وقت الإغلاق المستهدف** | 48 ساعة |

### المستوى 3: عالٍ (High)

| المعيار | الوصف |
|---|---|
| **النطاق** | 100-1000 مستخدم متضرر |
| **نوع البيانات** | أرقام هواتف + سجلات محادثات |
| **احتمال الضرر** | مرتفع — خطر سرقة هوية أو مضايقة أو ضرر نفسي |
| **المثال** | اختراق قاعدة بيانات Supabase وتسرب جزئي |
| **الإخطار الرقابي** | **إلزامي** خلال 72 ساعة |
| **إخطار المستخدمين** | **إلزامي** في أقرب وقت ممكن |
| **وقت الإغلاق المستهدف** | 24 ساعة (الاحتواء) |

### المستوى 4: حرج (Critical)

| المعيار | الوصف |
|---|---|
| **النطاق** | أكثر من 1000 مستخدم أو جميع المستخدمين |
| **نوع البيانات** | كامل قاعدة البيانات — أرقام، محادثات، بيانات أكاديمية |
| **احتمال الضرر** | حرج — ضرر جسيم محتمل على المستخدمين |
| **المثال** | اختراق كامل للنظام، هجوم ransomware، تسريب علني |
| **الإخطار الرقابي** | **إلزامي فوري** — الهدف 24 ساعة، حد أقصى 72 ساعة |
| **إخطار المستخدمين** | **فوري** — خلال ساعات |
| **تعليق الخدمة** | مطلوب إلى حين معالجة الثغرة |
| **وقت الإغلاق المستهدف** | 4 ساعات (الاحتواء الأولي) |

## 3. الاستجابة الفورية (الساعات 0-4)

### 3.1 اكتشاف الحادثة

**مصادر الاكتشاف المحتملة:**
- تنبيهات مراقبة Supabase / Vercel الآلية.
- إبلاغ من مستخدم لاحظ نشاطاً غير معتاد.
- اكتشاف داخلي من أحد المطورين أو فريق الأمن.
- إخطار من معالج خارجي (Supabase، Meta، Google).
- تقرير من باحث أمني خارجي.

### 3.2 خطوات الساعات الأربع الأولى

```
[الدقيقة 0] ← اكتشاف الحادثة
      │
      ▼
[الدقيقة 5] إخطار فوري للـ DPO ورئيس الهندسة عبر:
            - واتساب (قناة الطوارئ الداخلية)
            - الهاتف المباشر
      │
      ▼
[الدقيقة 15] فتح سجل حادثة جديد (Incident Log) بـ:
             - تاريخ ووقت الاكتشاف
             - طريقة الاكتشاف
             - الوصف الأولي
             - من اكتشف الحادثة
      │
      ▼
[الساعة 1] تقييم أولي للنطاق والخطورة:
           - كم مستخدماً متضرراً (تقدير)؟
           - ما نوع البيانات المعرضة؟
           - هل الاختراق مستمر أم متوقف؟
      │
      ▼
[الساعة 2] تصنيف الحادثة (1/2/3/4) واتخاذ قرار:
           - إذا المستوى 3 أو 4: تعليق الخدمة فوراً
           - إخطار المدير التنفيذي (CEO)
      │
      ▼
[الساعة 3] احتواء الحادثة:
           - عزل المكونات المتضررة
           - إلغاء مفاتيح API المشبوهة
           - تعطيل الحسابات المخترقة
           - تغيير كلمات المرور وبيانات الاعتماد
      │
      ▼
[الساعة 4] توثيق حالة الاحتواء الأولي
```

1. **التحقيق التفصيلي:** تحليل السجلات لتحديد نطاق الاختراق الدقيق.
2. **تقييم الأثر على الخصوصية (DPIA طارئ):** تحديد المخاطر الفعلية على المستخدمين.
3. **استمرار الاحتواء:** التأكد من إغلاق جميع نقاط الدخول.
4. **إعداد إخطار NCSA:** صياغة مسودة الإخطار الرسمي.
5. **إعداد إخطار المستخدمين:** صياغة رسالة واضحة وصادقة.
6. **جمع الأدلة الجنائية الرقمية:** نسخ السجلات بشكل صحيح لحفظ الأدلة.

## 7. مراحل التعافي والاستعادة

### 7.1 الساعات 24-72: التعافي

1. **تقييم شامل للنظام:** مراجعة جميع المكونات للتأكد من نظافتها.
2. **تحديث بيانات الاعتماد:** تغيير جميع كلمات المرور ومفاتيح API ورموز JWT.
3. **تعزيز الدفاعات:** تطبيق الإصلاحات والتحديثات الأمنية المطلوبة.
4. **اختبار آمنية النظام:** اختبارات اختراق موجّهة للثغرة المكتشفة.
5. **استعادة الخدمة تدريجياً:** بعد التأكد من الاحتواء الكامل.

### 7.2 الأيام 3-30: الاسترداد الكامل

1. إجراء تقييم شامل للمخاطر المتبقية.
2. تحديث DPIA بضوء الحادثة الجديدة.
3. مراجعة وتحديث سياسات الأمان.
4. تدريب الفريق على الدروس المستفادة.
5. تقديم تقرير نهائي مفصل للـ NCSA (إن طُلب).

## 8. مراجعة ما بعد الحادثة (Post-Incident Review)

يُعقد اجتماع مراجعة شامل خلال **7 أيام** من إغلاق الحادثة يضم:
- المدير التنفيذي (CEO)
- مسؤول حماية البيانات (DPO)
- رئيس الهندسة
- المستشار القانوني

**محاور الاجتماع:**

| المحور | الهدف |
|---|---|
| تحليل السبب الجذري (Root Cause Analysis) | فهم كيف حدث الاختراق |
| تقييم الاستجابة | ما الذي سار بشكل صحيح؟ وما الذي يحتاج إلى تحسين؟ |
| مراجعة الضوابط | هل الضوابط الموجودة كافية؟ |
| خطة الإجراءات التصحيحية | إجراءات محددة لمنع التكرار |
| مراجعة خطة الاستجابة | تحديث هذه الوثيقة إن لزم |

**مخرجات الاجتماع:**
- تقرير مراجعة موقّع من جميع المشاركين.
- خطة إجراءات تصحيحية مع مواعيد ومسؤوليات محددة.
- تحديث هذه الخطة (إصدار جديد).

## 9. سيناريوهات محددة للمشروع

### سيناريو أ: تسرب قاعدة بيانات Supabase
- **الاكتشاف:** تنبيه Supabase أو سجلات غير معتادة.
- **الاحتواء الفوري:** تعليق جميع مفاتيح API + تغيير كلمات مرور قاعدة البيانات.
- **التحقيق:** مراجعة سجلات الوصول (Supabase Audit Logs).
- **التصنيف المبدئي:** المستوى 3 أو 4.

### سيناريو ب: اختراق حساب WhatsApp Business
- **الاكتشاف:** رسائل غير مصرح بها أو تنبيه Meta.
- **الاحتواء الفوري:** إيقاف تشغيل Webhook + تعليق حساب WhatsApp.
- **التحقيق:** مراجعة سجلات Meta Business Manager.
- **التصنيف المبدئي:** المستوى 2 أو 3.

### سيناريو ج: كشف مفتاح API في كود مصدري (GitHub)
- **الاكتشاف:** تنبيه GitHub Secret Scanning أو تقرير باحث أمني.
- **الاحتواء الفوري:** إلغاء المفتاح المكشوف فوراً + إصدار مفتاح جديد.
- **التحقيق:** هل استُخدم المفتاح بشكل غير مصرح؟ مراجعة سجلات API.
- **التصنيف:** المستوى 1 (إذا لم يُستخدَم) أو 3/4 (إذا استُخدم).

### سيناريو د: هجوم Ransomware على البنية التحتية
- **الاكتشاف:** تعذّر الوصول إلى البيانات + رسالة طلب فدية.
- **الاحتواء الفوري:** عزل تام لجميع الأنظمة + عدم دفع الفدية.
- **الاحتواء:** التواصل مع Supabase/Vercel لاتخاذ إجراءات طارئة.
- **التصنيف:** المستوى 4 — إخطار NCSA فوري.
