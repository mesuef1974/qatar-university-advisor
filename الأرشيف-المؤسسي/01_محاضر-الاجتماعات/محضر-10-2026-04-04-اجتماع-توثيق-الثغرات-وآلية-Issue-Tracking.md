#### SEC-001 — Timing Attack على كلمة مرور Admin

| البند | التفاصيل |
|---|---|
| **الملف والسطر** | `api/admin.js:39` |
| **الشدة** | CRITICAL |
| **المشكلة** | استخدام عامل المقارنة `!==` العادية لمقارنة كلمة مرور المدير الإداري |
| **التأثير** | Timing Attack — يمكن لمهاجم متمرس قياس زمن الاستجابة وتخمين كلمة المرور حرفاً بحرف |
| **السبب الجذري** | مقارنة نصية عادية تستغرق وقتاً مختلفاً حسب عدد الأحرف المتطابقة، خلافاً للمقارنة الـ cryptographic المستغرقة لوقت ثابت |
| **الحل المطبّق** | استبدال المقارنة بـ `crypto.timingSafeEqual()` مع Buffer comparison — يضمن وقت استجابة ثابتاً بغض النظر عن صحة كلمة المرور |
| **التحقق** | تم اختبار الحل والتأكد من عمله بشكل صحيح |

**تعليق الأمن السيبراني:** هذه الثغرة كانت تُعرِّض لوحة الإدارة بالكامل للاختراق من أي جهة خارجية لديها صبر كافٍ لتنفيذ timing attack. الإصلاح فوري وضروري.

#### SEC-002 — Prompt Injection عبر تشفير المدخلات

| البند | التفاصيل |
|---|---|
| **الملف** | `lib/sanitizer.js` |
| **الشدة** | HIGH |
| **المشكلة** | `sanitizer.js` لا يكتشف Prompt Injection المشفّر عبر Base64 أو Unicode أو ROT13 أو غيرها |
| **التأثير** | يمكن لمستخدم خبيث تجاوز فلاتر الحماية بتشفير الأوامر الضارة، مما يتيح التلاعب بسلوك الـ AI |
| **السبب الجذري** | أنماط الكشف الأصلية تغطي النص الصريح (plaintext) فحسب، ولا تُفكّك أساليب التشفير قبل الفحص |
| **الحل المطبّق** | إضافة 8 أنماط encoding جديدة للكشف: base64، unicode escapes، rot13، hex encoding، unescape، decodeURI، btoa/atob patterns، String.fromCharCode |
| **التحقق** | تم اختبار جميع الأنماط الثمانية بنماذج حقيقية |

**تعليق الأمن السيبراني:** هذا النوع من الهجمات (Encoded Prompt Injection) من أكثر الأساليب استخداماً ضد أنظمة الـ AI. يُنصح بمراجعة دورية للأنماط مع كل تحديث.

#### SEC-003 — `unsafe-eval` في Content Security Policy

| البند | التفاصيل |
|---|---|
| **الملف والسطر** | `vercel.json:34` |
| **الشدة** | HIGH |
| **المشكلة** | إعداد CSP يحتوي على `'unsafe-eval'` في قسم `script-src` |
| **التأثير** | يُمكِّن تنفيذ `eval()` و`Function()` و`setTimeout(string)` — مما يفتح الباب لهجمات XSS في حال حقن أي سكريبت |
| **السبب الجذري** | إعداد CSP متساهل يعود لمرحلة التطوير الأولى ولم يُشدَّد قبل الإنتاج |
| **الحل المطبّق** | حذف `'unsafe-eval'` من `script-src` بالكامل |
| **التحقق** | تم التحقق من أن التطبيق لا يستخدم `eval()` في أي موضع — الحذف آمن |

**تعليق الأمن السيبراني:** `unsafe-eval` في CSP يُعدّ من أخطر الإعدادات إذ يُبطل جزءاً كبيراً من الحماية التي يوفرها CSP. يجب مراجعة إعدادات CSP بالكامل كل 6 أشهر.

#### SEC-004 — معرّف إخفاء الهوية قابل للتتبع الزمني (PDPPL)

| البند | التفاصيل |
|---|---|
| **الملف والسطر** | `api/cron-pdppl-cleanup.js:107` |
| **الشدة** | MEDIUM |
| **المشكلة** | معرّف إخفاء الهوية المُولَّد يتضمن `Date.now()` كجزء منه |
| **التأثير** | المعرّف يحمل معلومة زمنية قابلة للتتبع — انتهاك جزئي لمتطلبات قانون PDPPL القطري المتعلقة بإخفاء الهوية الحقيقي |

| **التأثير** | معلومات مضللة للمستخدمين الوافدين وغير القطريين — يُخل بجودة الإرشاد الأكاديمي الذي هو الغرض الأساسي من المشروع |
| **السبب الجذري** | (قيد التحقيق) — يُرجَّح أحد السببين أو كليهما: |
| | (أ) عدم وجود حقل `nationality_filter` أو ما يُعادله في بيانات المنح |
| | (ب) الـ prompt المرسل للـ AI لا يُضمِّن شرط الجنسية عند صياغة الاستجابة |

**نقاش البدائل الهندسية:**

أشار مدير الهندسة إلى ثلاثة مسارات محتملة للإصلاح:

| ب — فلتر في البيانات | إضافة عمود `eligibility` لكل منحة في قاعدة البيانات مع قيم (قطريون / غير قطريين / الجميع) | أكثر دقة ويوصى به |
| ج — الجمع بين أ و ب | تطبيق كلا المسارين معاً | الأفضل على المدى البعيد |

**توصية الهندسة:** تبني المسار (ج) في Sprint القادم كـ HIGH priority.

### القسم الثالث: QA المتقدم — مقترح نظام Issue Tracking الرسمي

**المقدِّم:** مدير ضمان الجودة المتقدم

عرض مدير QA أن المشروع بلغ مرحلة نضج تستوجب تبني نظام توثيق رسمي للمشاكل، مقدِّماً المقترح التالي:

#### مقترح نظام Issue Tracking

**المكونات الأساسية:**

1. **سجل ثغرات رسمي** — ملف Markdown مركزي يحتوي على جميع المشاكل المكتشفة تاريخياً وحاضراً.

2. **تصنيف موحَّد للشدة (Severity):**
   - `CRITICAL` — يؤثر مباشرة على أمن النظام أو يُوقف الخدمة
   - `HIGH` — تأثير كبير على الوظائف الجوهرية أو الأمن
   - `MEDIUM` — تأثير متوسط أو مخالفة لمتطلبات غير جوهرية
   - `LOW` — تحسينات أو مشاكل مستحسَن إصلاحها

3. **تصنيف موحَّد للأولوية (Priority):**
   - `P0` — يُوقف Sprint الحالي — إصلاح فوري
   - `P1` — يُدرَج في Sprint القادم حتماً
   - `P2` — يُدرَج في أقرب Sprint مناسب
   - `P3` — Backlog — يُعالَج عند توفر الوقت

4. **دورة حياة موحَّدة للـ Issue:**
   ```
   Open → In Progress → Fixed → Verified → Closed
   ```

5. **التزام 24 ساعة:** أي مشكلة مكتشفة يجب توثيقها في السجل خلال 24 ساعة من الاكتشاف.

**تعليق CEO:** المقترح مقبول بالكامل — يُعتمد اعتباراً من اليوم.
