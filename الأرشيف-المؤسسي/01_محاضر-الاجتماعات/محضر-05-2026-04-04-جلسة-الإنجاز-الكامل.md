### 3.2 — T-009: إصلاح In-Memory Profile (Vercel Multi-Instance)

**الملفات المعدّلة:**
- `lib/supabase.js` — دوال جديدة
- `lib/user-profiler.js` — TTL cache
- `lib/findResponse.js` — pipeline جديد

**المشكلة:**
Vercel Serverless يُنشئ instances متعددة، كل instance له ذاكرة مستقلة. المستخدم يتحدث مع instance مختلف في كل رسالة → profile يُفقَد.

**الحل المُنفَّذ:**

```
المستخدم يُرسل رسالة
    ↓
getInMemoryProfile(phone) — هل الكاش صالح؟ (TTL: 30 دقيقة)
    ↓ لا              ↓ نعم
Promise.all([          استخدم الكاش
  getOrCreateUser(),   مباشرة
  getUserProfileData()
])
    ↓
Supabase = مصدر الحقيقة الوحيد
    ↓
احفظ في Kاش المحلي
    ↓
buildUserProfile() — تحديث من رسالة جديدة
    ↓
هل تغيّر الملف؟
    ↓ نعم
saveUserProfileData() → Supabase (غير متزامن)
saveInMemoryProfile() → كاش محلي (فوري)
```

**الدوال الجديدة في `lib/supabase.js`:**

```javascript
// حفظ الملف الكامل كـ JSONB
async function saveUserProfileData(phone, profileData)

// استرجاع الملف مع دمج الأعمدة المنفصلة
async function getUserProfileData(phone)
```

**التحسينات في `lib/user-profiler.js`:**
```javascript
const CACHE_TTL_MS = 30 * 60 * 1000; // 30 دقيقة

// getInMemoryProfile: يُرجع null عند انتهاء الصلاحية
// saveInMemoryProfile: يحفظ مع timestamp
// invalidateInMemoryProfile: يمسح الكاش فوراً
```

```
الحالات الثلاث:

CLOSED (طبيعي) ─────── 3 فشل متتالي ──────► OPEN (مُغلَق)
                                                    │
                                            30 ثانية انتظار
                                                    │
CLOSED ◄─── 2 نجاح ─── HALF_OPEN (اختبار) ◄────────┘
```

**الإعدادات:**
```javascript
const CONFIG = {
  FAILURE_THRESHOLD: 3,      // عدد الفشل قبل الفتح
  RESET_TIMEOUT_MS:  30_000, // 30 ثانية قبل المحاولة التجريبية
  SUCCESS_THRESHOLD: 2,      // نجاحات لإغلاق الدائرة
};
```

**التكامل في `lib/supabase.js`:**
```javascript
// كل دالة Supabase مُغلَّفة بـ _withCB()
async function _withCB(fn, fallback = null) {
  return supabaseCircuit.execute(fn, () => fallback);
}

// عند فشل Supabase:
// - getUserProfileData() → {} (يبدأ من صفر)
// - getConversationHistory() → [] (بدون سياق)
// - saveMessage() → silent fail (لا يتوقف البوت)
```

### 3.4 — T-BUG-001: إصلاح منح غير القطريين

**الملف:** `lib/findResponse.js`

**البلاغ من العميل:**
> "يوجد بعض الاجابات غير المنطقية مثلا منح غير القطريين ويجيب بمنح القطري والمنحة الاميرية"

**تشخيص المشكلة — 3 أخطاء:**

| # | الخطأ | السبب |
|---|-------|-------|
| 1 | `q.includes('غير قطري')` لا يتطابق مع "غير القطريين" | "ال" التعريف + صيغة الجمع |
| 2 | catch-all `منح → scholarship_amiri` يأتي قبل فحص غير القطريين | ترتيب خاطئ في الكود |
| 3 | لا يوجد فحص جنسية الملف الشخصي كطبقة حماية إضافية | منطق ناقص |

**مسار "منح غير القطريين" قبل الإصلاح:**
```
"منح غير القطريين"
    ↓
isNonQatariKeyword? → false (لأن 'غير القطريين' ≠ 'غير قطري')
    ↓
catch-all: q.includes('منح') → TRUE
    ↓
❌ scholarship_amiri  ← خطأ!
```

**مسار "منح غير القطريين" بعد الإصلاح:**
```
"منح غير القطريين"
    ↓
isNonQatariKeyword? → TRUE (يشمل 'غير القطريين')
    ↓
isScholarshipContext? → TRUE (يشمل 'منح')
    ↓
✅ scholarship_non_qatari  ← صحيح!
```

**الكلمات المُضافة للفحص:**
```javascript
const isNonQatariKeyword =
  q.includes('غير قطري') || q.includes('غير القطري') || q.includes('غير القطريين') ||
  q.includes('مقيم') || q.includes('للمقيمين') || q.includes('مقيمين') ||
  q.includes('وافد') || q.includes('وافدين') || q.includes('للوافدين') ||
  q.includes('أجنبي') || q.includes('أجانب') || q.includes('للأجانب') ||
  q.includes('منح دولية') || q.includes('لغير القطريين');
```

**طبقة الحماية الثالثة (في processMessage):**
```javascript
// إذا وصل scholarship_amiri لمستخدم non_qatari → تصحيح تلقائي
if (result.key === 'scholarship_amiri' && profile.nationality === 'non_qatari') {
  result.key = 'scholarship_non_qatari';
}
```
