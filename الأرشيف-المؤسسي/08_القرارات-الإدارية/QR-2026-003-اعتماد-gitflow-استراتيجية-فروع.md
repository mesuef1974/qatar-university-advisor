## نص القرار الرسمي

> **يُعتمد نموذج GitFlow رسمياً** كاستراتيجية إدارة فروع مشروع Qatar University Advisor بدءاً من 2026-04-04، مع تطبيق Semantic Versioning وConventional Commits كمعايير إلزامية لجميع المساهمين في الكود.

---

## هيكل الفروع المعتمد

```
main          ← الإنتاج — محمي — موافقة CEO إلزامية
  │
develop       ← التكامل/Staging — محمي — موافقة CEO إلزامية
  │
  ├── feature/[اسم-الميزة]     ← ميزات جديدة
  ├── fix/[اسم-الإصلاح]        ← إصلاح أخطاء عادية
  ├── hotfix/[اسم-الطارئ]      ← إصلاح طارئ يذهب لـ main مباشرة
  └── release/v[X.Y.Z]         ← تجهيز إصدار رسمي
```

---

## نظام الإصدارات (Semantic Versioning)

| النوع | متى | مثال |
|-------|-----|------|
| PATCH (v1.0.X) | إصلاح أخطاء — security fix | v1.0.1 |
| MINOR (v1.X.0) | ميزة جديدة متوافقة | v1.1.0 |
| MAJOR (vX.0.0) | تغيير جذري — قد يكسر التوافق | v2.0.0 |

---

## Conventional Commits — معيار إلزامي

```
النوع(النطاق): الوصف المختصر

feat:     ميزة جديدة
fix:      إصلاح خطأ
sec:      إصلاح أمني
docs:     توثيق فقط
chore:    صيانة (dependencies, configs)
ci:       تغييرات CI/CD Pipeline
release:  إصدار جديد
```

---

## القواعد الذهبية

| القاعدة | السبب |
|---------|-------|
| لا `git push --force` على أي فرع محمي | يُتلف التاريخ ويُضيع عمل الزملاء |
| لا commit مباشر على `main` أو `develop` | يتجاوز المراجعة — خطر على الإنتاج |
| كل merge لـ `main` يصحبه git tag فوري | تتبع الإصدارات لكل deployment |
| `npm ci` بدلاً من `npm install` في CI | reproducible builds — حماية من drift |
